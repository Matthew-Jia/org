# Modes

## Real Mode (1978)
When the ``Reset" signal hits the CPU, it enters this mode. You are limited to 16-bit registers (AX, BX, etc.). You can only see 1MB of RAM. There is no concerpt of ``User" or ``Kernel"-any code can overwrite the BIOS data or the interrupt table. This was how the oldest operating systems running on x86-64 started (and operated in), and to maintain backwards compatibility (allowing old OS to run on new hardware), modern x86-64 processors still start in this mode.

## Protected Mode (1985)
To break the 1MB barrier, you switch to Protected Mode. This is where the GDT was born. After you load a GDT and set the PE (protection Enable) bit in the CR0 register, you can address 4GB. Here, the GDB provides "Segments" to protect the Kernel from User apps.

## Long Mode (2003)
This is where our kernel lives. To get here, you must first be in Protected Mode. You enable Paging, set the LME (Long Mode Enable) bit in the EFER register, and perform a Far Jump (a function return and a segment change at the same time) using a GDT selector that has the ``L" (Long) bit set to 1. You can directly jump from Real Mode to Long Mode by setting up the paging tables and the GDT to 64 bit segments and performning a long jump, but this can still be seen as going from Real Mode $\rightarrow$ Protected Mode since Long Mode is a sub-mode of Protected Mode (you still need the PE bit enabled, you just don't need to run any 32 bit instructions).

# Memory Layout

On 64-bit systems, only 48 bits of virtual address space are given to a user program. Additionally, bits 48-64 are always whatever bit 47 is (sign extended). This creates a "lower half" and "higher half" partition of the address space with a very large gap between them. The the top 2GB of every processes' virtual address are mapped to the kernel. This means bits 32-64 are always 1 when accessing kernel code. There are a couple of benefits of laying out memory like this. 

- Isolates kernel memory from user memory. Kernel memeory and user memory are very far apart.
- This keeps the kernel in the same location in memory for every process, and ensure all processes start in the location in virtual memory that they expect (at the beginning). 
- When the OS switches from running "Process A" to "Process B", it has to swap out the page tables. However, if the kernel is mapped in the high half of every process's page table, the kernel page mappings can be kept between context switches.
- By not mapping the entire upper half of virtual address space to the kernel and only using the last 2 GB, you can use 32 bit arithmatic and sign extend any kernel pointers to remake the original kernel address. This allows for the binary to be smaller
